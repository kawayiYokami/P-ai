# 20260420_工具类型重做计划

## 背景

当前工具系统仍以“普通系统工具 + 部门权限过滤 + 少量隐藏特判”为主，已经无法准确表达下面几类工具的真实语义：

1. 永远存在、不可关闭、应视为系统固定能力的工具
2. 仅在本地会话中存在、且不可关闭的工具
3. 仅在联系人会话中存在、目标固定为当前联系人的专用隐藏工具

现状问题主要有：

- `todo / remember / recall / plan` 仍按普通 builtin 工具处理
- `plan` 仍受部门工具权限影响
- `remote_im_send` 仍作为单一普通工具存在，协议复杂，语义混乱
- 前端工具目录与后端运行时工具装配，都还没有“工具类型”这一层稳定模型

本次要做的是“工具类型重做”，先把工具类型模型立住，再按类型重挂载具体工具。

## 本次目标

建立稳定的三种工具类型，并让前后端都按工具类型工作，而不是继续靠零散特判。

目标类型如下：

### 1. 固定系统工具

这类工具：

- 永远可用
- 不可关闭
- 不受部门权限控制
- 既可出现在普通会话，也可出现在联系人会话
- 仍属于系统能力工具的一部分
- 不出现在部门权限页

当前归入此类的工具：

- `todo`
- `remember`
- `recall`

### 2. 本地会话固定工具

这类工具：

- 永远可用
- 不可关闭
- 不受部门权限控制
- 仅在“本地会话”中注入
- 联系人会话中完全不注入
- 不出现在部门权限页

当前归入此类的工具：

- `plan`

### 3. 联系人专用隐藏工具

这类工具：

- 只在“联系人会话”中注入
- 永远不受部门权限控制
- 永远不出现在系统工具展示页
- 永远不允许模型自行选择联系人、渠道或目标
- 发送目标固定为“当前联系人”

本次将 `remote_im_send` 拆成 3 个工具：

- `contact_reply`
- `contact_send_files`
- `contact_no_reply`

其中：

- `contact_reply(text)`：立即给当前联系人发一句话
- `contact_send_files(file_paths)`：立即给当前联系人发附件
- `contact_no_reply(reason?)`：声明本轮结束时不要自动发送最终回复，并记录原因供后续轮次参考

## 范围边界

本次只重做“工具类型模型”和“相关工具挂载规则”，不顺带做以下扩展：

- 不重做部门权限卡的 UI 结构
- 不重做 MCP 工具权限模型
- 不重做普通系统工具的文案体系
- 不改联系人自动回复状态机的大逻辑（除为新工具协议做必要接线）

## 现状与问题

### 1. 运行时工具装配仍按普通工具判断

当前 `assemble_runtime_tools(...)` 里：

- `todo / remember / recall / plan` 仍走 `tool_enabled(...)`
- `remote_im_send` 通过“联系人会话强制启用”做特判

这会导致：

- 不可关闭工具仍被当成“可配置工具”
- `plan` 仍会被联系人会话看到，或被部门过滤掉
- 联系人工具语义继续依赖旧的 `remote_im_send`

### 2. 前端 builtin tools 目录仍是普通列表

当前前端默认工具表里，以下工具仍被当成普通可配置项：

- `remember`
- `recall`
- `plan`
- `todo`
- `remote_im_send`

这与目标模型冲突。

### 3. 系统提示词中的工具规则也还未完全按新类型切换

当前 `prompt_manager.rs` 中：

- `remember / recall` 的规则注入仍受部门工具可用性判断影响
- `plan` 的规则注入也仍受部门判断影响

而按本次新模型：

- `todo / remember / recall` 应视为固定系统工具
- `plan` 应视为本地会话固定工具
- 联系人工具的说明需要切换到新三工具协议

## 重构方案

### 一、引入稳定的工具类型模型

后端与前端都统一引入工具类型概念，至少要能表达：

- `always_on`
- `always_on_local_conversation`
- `contact_only_hidden`
- `department_controlled`

其中：

- `todo / remember / recall` → `always_on`
- `plan` → `always_on_local_conversation`
- `contact_reply / contact_send_files / contact_no_reply` → `contact_only_hidden`
- 其他系统工具 → `department_controlled`

本次不要求一定暴露为对外 API 枚举字段，但运行时与展示层必须共享这一语义。

### 二、重写联系人工具协议

删除旧的 `remote_im_send` 运行时入口，改为 3 个联系人专用工具：

- `contact_reply`
- `contact_send_files`
- `contact_no_reply`

规则如下：

- 仅联系人会话注入
- 普通会话永不注入
- 不受部门权限控制
- 不出现在工具页
- 不出现在部门权限页
- 发送目标固定为当前联系人

自动回复规则：

- 若本轮未调用 `contact_no_reply`，则轮次结束时自动把最终 assistant 文本发给当前联系人
- `contact_reply` 与 `contact_send_files` 只是中途动作，不会取消最终自动回复
- 只有 `contact_no_reply` 会取消本轮最终自动回复

### 三、重写运行时工具装配逻辑

`assemble_runtime_tools(...)` 需要从“一个个问 `tool_enabled`”改成“先按工具类型决定，再按普通工具补部门权限”。

目标行为：

- `todo / remember / recall`：始终挂载
- `plan`：仅本地会话挂载
- 联系人三工具：仅联系人会话挂载
- 其他普通系统工具：继续按部门权限决定

同时要清理旧的：

- `remote_im_send` 强制注入逻辑
- `remote_im_send` 的 manifest 与 unavailable reason 逻辑

### 四、重写系统提示词中的工具规则注入

工具规则必须和新工具类型一致：

- `todo / remember / recall` 的规则默认存在
- `plan` 工具规则仅在本地会话注入
- 联系人会话不再注入 `plan` 工具规则
- 联系人会话改为注入 `contact_reply / contact_send_files / contact_no_reply` 的规则说明
- 旧的 `remote_im_send` 说明块删除

### 五、重写前端工具目录与工具展示

工具页与权限页要按新模型更新：

- `todo / remember / recall / plan` 不再作为普通可开关工具理解
- `todo / remember / recall` 不出现在部门权限列表
- `plan` 不出现在部门权限列表
- 联系人三工具不出现在系统工具展示页
- 联系人三工具不出现在部门权限列表
- 普通可控工具列表只展示真正 `department_controlled` 的工具

如果需要保留目录中的“固定工具说明”，也必须以只读说明方式呈现，不能做成可切换开关。

## 实现步骤

### 阶段 1：定义工具类型

1. 梳理后端 builtin tool 的统一分类入口
2. 梳理前端 builtin tool 默认目录的统一分类入口
3. 为 `todo / remember / recall / plan / remote_im_send` 迁移到新类型预留结构

### 阶段 2：替换联系人工具协议

1. 新增 `contact_reply`
2. 新增 `contact_send_files`
3. 新增 `contact_no_reply`
4. 删除 `remote_im_send` 运行时挂载与提示词说明
5. 保持自动最终回复逻辑不变，只改中途控制协议

### 阶段 3：重写运行时工具挂载

1. 运行时工具装配按工具类型分流
2. 联系人会话只注入联系人专用隐藏工具
3. 本地会话固定注入 `plan`
4. 固定系统工具从部门权限判断中移除

### 阶段 4：重写前端展示和权限来源

1. 工具页排除联系人专用隐藏工具
2. 部门权限页排除固定工具与联系人专用工具
3. 普通部门可控工具列表只保留真正受部门控制的系统工具

### 阶段 5：回归与清理

1. 更新相关测试
2. 更新工具文案与说明
3. 更新 `CHANGELOG.md`
4. 删除旧 `remote_im_send` 遗留逻辑与测试

## 测试与验证

至少覆盖以下场景：

### 运行时挂载

- 普通会话中：
  - 有 `todo / remember / recall / plan`
  - 没有 `contact_reply / contact_send_files / contact_no_reply`

- 联系人会话中：
  - 有 `todo / remember / recall`
  - 没有 `plan`
  - 有 `contact_reply / contact_send_files / contact_no_reply`

### 权限过滤

- 部门白黑名单变化时：
  - 不影响 `todo / remember / recall`
  - 不影响 `plan`
  - 不影响联系人三工具
  - 只影响真正 `department_controlled` 的系统工具

### 系统提示词

- 本地会话中包含 `plan` 工具规则
- 联系人会话中不包含 `plan` 工具规则
- 联系人会话中包含新三工具规则
- 系统提示词中不再出现 `remote_im_send`

### 联系人自动回复

- 联系人会话中未调用任何联系人专用工具时：
  - 仍自动发送最终 assistant 回复

- 调用 `contact_reply` 后：
  - 先发送即时回复
  - 轮次结束时仍自动发送最终 assistant 回复

- 调用 `contact_send_files` 后：
  - 附件立即发出
  - 轮次结束时仍自动发送最终 assistant 回复

- 调用 `contact_no_reply` 后：
  - 本轮结束时不自动发送最终 assistant 回复
  - `reason` 被记录，可供后续轮次使用

## 风险点

### 1. 工具目录与运行时挂载分叉

如果前端仍保留旧工具列表，而后端已改成新类型，工具页和实际可用工具会再次不一致。

### 2. 联系人自动回复与中途 reply 重复发送

如果工具循环结束后的自动发送逻辑没有正确识别 `contact_no_reply`，容易出现：

- 先 `contact_reply`
- 再自动发最终回复

这本身是允许的；但如果又误把 `contact_reply` 当成“最终决定”，会出现逻辑冲突，需要把语义定死。

### 3. 旧 `remote_im_send` 历史兼容

部分测试、提示词、工具循环停止条件、历史消息解析，可能仍依赖旧工具名，需要完整迁移。

## 本次完成判定

满足以下条件时，可视为本次计划完成：

1. 工具系统能稳定表达三种工具类型
2. `todo / remember / recall` 成为固定系统工具
3. `plan` 成为仅本地会话注入的固定工具
4. `remote_im_send` 被新三工具替换
5. 联系人三工具不受部门权限控制，也不出现在工具展示/权限页面
6. 系统提示词、运行时工具挂载、前端展示三处口径一致
