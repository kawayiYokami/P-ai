# 20260427_续调度激活语义修复计划

## 1. 背景

当前聊天续调度链路出现严重竞态：一轮结束后，消息出队、任务提醒、计划执行后的上下文整理等后台入口可能需要再次激活会话部门，但前端只有“用户主动发送”路径能稳定生成助理草稿气泡。

已观察到的问题包括：

1. 消息显示错乱，新的流式轮次可能带上上一条回复残留。
2. 后台续调度有时无法激活 LLM。
3. 计划执行触发 `organize_context` / 上下文整理后，没有自动再次激活 LLM。
4. 用户发送路径正常，但任务提醒、消息出队、压缩后续跑等后台激活路径不稳定。

本计划目标是收口“激活语义”，让所有会话部门激活都经过统一协议，并由该协议驱动前端草稿气泡生成。

## 2. 当前事实

### 2.1 后端调度主链路

`src-tauri/src/features/chat/scheduler.rs` 中，`process_conversation_batch` 当前主顺序是：

1. 批量事件写入正式历史：`commit_scheduler_history_flush(...)`
2. 广播 `easy-call:history-flushed`
3. 根据 `event_activate_flags` 判断 `should_activate`
4. 若需要激活，调用 `activate_main_assistant(...)`
5. 激活前广播 `easy-call:round-started`
6. 流式完成后广播 `easy-call:round-completed` 或 `easy-call:round-failed`

这个“先写历史，再激活助理”的方向是正确的，不能回退。

### 2.2 当前 `round_started` 事件信息不足

当前 `emit_round_started_event(...)` payload 只有：

```json
{
  "conversationId": "..."
}
```

它缺少：

1. 本次激活唯一标识。
2. 激活来源。
3. 执行部门与人格。
4. 是否来自用户发送、任务提醒、消息出队、上下文整理后续跑、远程联系人待办续跑等语义。

因此前端无法把“某次后台激活”稳定映射成一轮新的前台草稿。

### 2.3 前端草稿生成入口不统一

`src/features/chat/composables/use-chat-flow.ts` 当前有两套路径：

1. 用户发送路径：
   - `sendChat()` 创建 `gen`
   - 插入用户乐观草稿
   - 设置 `round = queued`
   - 创建助理预流式草稿
   - 收到 `round_started` 后进入 waiting / streaming

2. 后台事件路径：
   - `handleExternalHistoryFlushed(...)` 收到 `activateAssistant=true` 时会 `++generation`
   - 但没有稳定进入 `queued` 状态
   - `handleExternalRoundStarted(...)` 只有在 `round.phase === "queued"` 或 `sendChatActiveGen` 存在时才有效

结果：后台激活收到 `round_started` 时，前端可能没有对应 round，自然无法生成助理草稿。

### 2.4 流式缓存清理不完整

`use-chat-flow.ts` 维护了 `conversationStreamCache`，用于流式中切会话、重绑通道后恢复草稿。

当前问题：

1. 缓存按 `conversationId` 存储，但不区分轮次。
2. 完成、失败、停止路径有清理，但依赖当前 `generation` / 当前 round 状态。
3. 后台激活若没进入正确 round，完成事件可能无法走到完整清理路径。
4. 下一轮激活前如果没有先清理旧缓存，可能出现“带上上一条”的显示污染。

### 2.5 上下文整理后续激活语义缺失

`src-tauri/src/features/system/commands/archive_pipeline.rs` 的 `emit_compaction_history_flushed_event(...)` 当前发送：

```json
{
  "activateAssistant": false,
  "compactionApplied": true
}
```

`src-tauri/src/features/chat/model_runtime/provider_and_stream/tool_loop.rs` 中，`organize_context` 成功后会返回：

```rust
suppress_assistant_message: true
```

并且 `builtin_organize_context(...)` 只是调用：

```rust
trigger_chat_queue_processing(app_state);
```

这意味着上下文整理只表达“历史被压缩刷新”，没有明确表达“压缩后继续激活当前部门回答”。

所以“计划执行压缩上下文之后没有自动再次激活 LLM”不是偶发，而是当前协议没有这个语义。

## 3. 目标

1. 所有会话部门激活都走统一入口。
2. 前端助理草稿气泡由“激活事件”驱动，而不是只由用户发送路径隐式创建。
3. 后台激活能和用户发送一样进入 queued / waiting / streaming 状态。
4. 流式缓存按轮次收口，完成、失败、停止后不污染下一轮。
5. 上下文整理后需要继续回答时，有明确的 follow-up 激活语义。

## 4. 非目标

1. 不重写整个聊天调度器。
2. 不改变“先写正式历史，再激活助理”的核心顺序。
3. 不重新设计远程 IM 状态机。
4. 不扩展新的 UI 面板。
5. 不把子委托线程接入前台流式轨道。

## 5. 激活协议设计

### 5.1 扩展 `round_started` payload

保留现有事件名 `easy-call:round-started`，降低前后端接线风险；扩展 payload：

```json
{
  "conversationId": "conversation-xxx",
  "activationId": "activation-xxx",
  "requestId": "request-xxx",
  "reason": "queue_dispatch",
  "departmentId": "assistant-department",
  "agentId": "default-agent"
}
```

字段说明：

| 字段 | 说明 |
|---|---|
| `conversationId` | 被激活的会话 |
| `activationId` | 本次激活唯一标识，用于前端区分轮次 |
| `requestId` | 模型请求链路追踪 ID，可复用 `RuntimeContext.request_id` |
| `reason` | 激活来源 |
| `departmentId` | 本次执行部门 |
| `agentId` | 本次执行人格 |

### 5.2 激活来源枚举

首轮只需要覆盖现有入口：

| reason | 语义 |
|---|---|
| `user_send` | 用户主动发送后激活 |
| `queue_dispatch` | 普通队列消息出队后激活 |
| `task_due` | 任务提醒触发激活 |
| `guided_queue` | 引导投送续调度激活 |
| `remote_im_followup` | 远程联系人待办续跑激活 |
| `context_compaction_followup` | 上下文整理完成后继续回答 |
| `unknown` | 暂时无法归因的兼容来源 |

### 5.3 completed / failed 对齐

后续 `round_completed` / `round_failed` 也应携带同一个 `activationId`。

最小实现允许第一阶段只扩 `round_started`，但缓存清理要优先按 `conversationId` 保底；若改动面可控，应同步扩 completed / failed。

## 6. 前端状态机调整

### 6.1 统一入口

新增内部函数，名称建议：

```ts
beginAssistantActivationFromEvent(payload)
```

职责：

1. 校验 `conversationId` 是否匹配当前前台会话。
2. 为本次激活建立前端 round。
3. 清理当前会话旧 `conversationStreamCache`。
4. 生成或更新助理草稿。
5. 设置 `frontendRoundPhase = "waiting"`。
6. 设置 `chatting = true`。

### 6.2 用户发送路径

用户发送仍保留用户乐观草稿。

但助理草稿不再只依赖 `sendChat()` 自己创建，而应接受 `round_started` 统一推进。

兼容策略：

1. `sendChat()` 仍可进入本地 `queued`，显示“正在准备”。
2. 收到 `round_started` 后，用 activation payload 绑定本地 queued round。
3. 如果本地没有 queued round，但当前会话匹配，也要创建后台激活 round。

### 6.3 后台激活路径

`handleExternalHistoryFlushed(...)` 不再承担“隐式创建可流式 round”的职责。

它只负责：

1. 合并已落库消息。
2. 保留 `activateAssistant` 作为提示信息。
3. 不直接决定草稿生成。

真正草稿生成由 `handleExternalRoundStarted(...)` 统一处理。

### 6.4 流式 delta 路径

收到普通 `assistant-delta` 时：

1. 若已有当前 activation 的 streaming round，直接追加。
2. 若没有 round 但事件匹配当前会话，允许按缓存恢复草稿。
3. 恢复前必须确认不是旧 activation 残留；如果事件没有 activationId，则只能按“当前会话 runtimeState=assistant_streaming”保底恢复。

## 7. 流式缓存清理规则

### 7.1 开始新激活前清理

收到 `round_started` 且匹配当前会话时：

1. 清理该会话旧 `conversationStreamCache`。
2. 清空 `latestAssistantText` / reasoning / tool status。
3. 清空旧 `streamToolCalls`。
4. 移除旧 assistant draft。

### 7.2 轮次结束后清理

以下路径都必须清理缓存：

1. `round_completed`
2. `round_failed`
3. 用户 stop 成功
4. queued 阶段 stop
5. 后端返回 `CHAT_ABORTED_BY_USER`
6. 前台切会话冻结并最终完成时，至少清目标会话缓存

### 7.3 缓存键升级

可选但推荐：

```ts
conversationStreamCache: Map<conversationId, {
  activationId?: string;
  ...streamState
}>
```

读取缓存时必须满足：

1. `conversationId` 匹配。
2. 如果双方都有 `activationId`，必须相同。
3. 如果没有 `activationId`，只能在 `runtimeState === "assistant_streaming"` 时保底使用。

## 8. 上下文整理后续跑策略

### 8.1 区分两个事件

`history_flushed.compactionApplied=true` 只表示：

1. 上下文整理消息已写入历史。
2. 前端需要刷新消息列表。
3. 旧的流式/草稿状态应被收口。

它不应该同时表达是否继续激活。

### 8.2 `organize_context` 成功后的 follow-up

当模型调用 `organize_context` 成功后，当前调度不应该就此静默结束，而应明确触发一次后续激活：

```text
context_compaction_followup
```

可选实现路线：

1. 在 `organize_context` 成功后向当前会话插入一个内部 pending event，`activate_assistant=true`。
2. 或在当前调度收口处直接调用统一激活入口。

优先建议第一种：继续复用“队列事件先写历史，再激活”的主调度语义。

### 8.3 避免重复激活

需要保证：

1. 同一个 `organize_context` 成功只触发一次 follow-up。
2. 如果当前会话已经有 guided queue，则优先走 guided queue 的既有续调度，不重复生成两个 LLM 轮次。
3. 如果上下文整理失败，不触发 follow-up。

## 9. 后端改动点

### 9.1 `chat/scheduler.rs`

1. 扩展 `emit_round_started_event(...)` 参数。
2. 从 `RuntimeContext` / `ChatSessionInfo` 组装 `activationId`、`reason`、`departmentId`、`agentId`。
3. 在 `activate_main_assistant(...)` 内统一发扩展后的 `round_started`。
4. `emit_round_completed_event(...)` / `emit_round_failed_event(...)` 尽量同步携带 `activationId`。
5. 远程联系人 follow-up 循环也要传 reason：`remote_im_followup`。

### 9.2 `task/scheduler.rs`

任务提醒已有 `RuntimeContext`：

```rust

```

需要确保该语义能映射到 `reason=task_due`。

### 9.3 `archive_pipeline.rs` / `builtin_runtime_context.rs`

预计改动：

1. 保留 `compactionApplied=true` 的 history refresh。
2. 增加上下文整理成功后的 follow-up 触发语义。
3. 避免只 `trigger_chat_queue_processing(...)` 但没有新 pending event 的空转。

### 9.4 `tool_loop.rs`

`organize_context_succeeded(...)` 当前直接返回 suppress assistant message。

需要确认：

1. 返回前是否已经安排 follow-up。

## 11. 验收场景

### 11.1 用户普通发送

1. 用户消息先出现。
2. 助理草稿随后出现。
3. 流式正常追加。
4. 完成后草稿转正式消息。
5. 下一轮不带上一轮残留。

### 11.2 任务提醒激活

1. 到点任务写入会话。
2. 前端收到 history flush 后显示任务消息。
3. 收到 round started 后自动生成助理草稿。
4. LLM 自动开始流式。

### 11.3 消息出队激活

1. 后台消息出队写入正式历史。
2. 前台当前会话能看到消息。
3. 若 `activateAssistant=true`，自动出现助理草稿并开始流式。

### 11.4 上下文整理后续跑

1. `organize_context` 成功写入 `[上下文整理]` 消息。
2. 前端刷新整理消息。
3. 随后收到 `context_compaction_followup` 激活。
4. 自动生成新助理草稿。
5. LLM 基于整理后的上下文继续回答。

### 11.5 流式缓存不串轮

1. 连续两轮回复。
2. 第二轮草稿不带第一轮文本。
3. 第二轮工具状态不带第一轮工具调用。
4. 切会话再切回不恢复已完成轮次的旧缓存。

## 13. 风险与约束

1. 聊天调度是核心链路，禁止用“只改 generation”这类局部补丁掩盖状态机问题。
2. 保持“先写历史，再激活助理”不变。
3. 前端草稿只能由统一激活语义创建或推进，避免用户路径和后台路径继续分裂。
4. 完成、失败、停止都必须清理流式缓存。
5. 上下文整理后续跑必须显式表达，不能靠 `trigger_chat_queue_processing` 空转。

## 14. 实施顺序

1. 扩展后端 `round_started` payload，并尽量同步 completed / failed 的 activation 信息。
2. 前端接入统一 activation 入口，保证后台激活可生成草稿。
3. 加强流式缓存清理，先解决“带上上一条”。
4. 修正 `organize_context` 成功后的 follow-up 激活。
5. 跑最小验证。
6. 根据实际 diff 更新本计划为实现回顾，再提交。

## 15. 当前结论

本问题的根因是激活协议不完整，而不是某一个 UI 气泡漏插。

正确修复方向：后端把“会话部门被激活”作为一等事件广播；前端只认这个统一事件创建助理草稿；流式缓存按轮次收口；上下文整理后的继续回答用明确 follow-up 激活表达。

### 16.1 后端激活协议

`src-tauri/src/features/chat/scheduler.rs` 已将主助理激活结果收口为 `ActivatedAssistantResult`，包含：

1. `SendChatResult`
2. `activation_id`
3. `request_id`

`easy-call:round-started` 已扩展 payload：

```json
{
  "conversationId": "...",
  "activationId": "...",
  "requestId": "...",
  "reason": "...",
  "departmentId": "...",
  "agentId": "..."
}
```

`round_completed` / `round_failed` 已同步携带 `activationId` / `requestId`，用于前端按轮次收口缓存。

### 16.2 激活来源

当前 `RuntimeContext.dispatch_reason` 映射到以下激活来源：

1. `user_send`
2. `task_due`
3. `remote_im_followup`
4. `context_compaction_followup`
5. `guided_queue`
6. `delegate_continue`
7. 兜底 `queue_dispatch`

远程联系人待办续跑已显式构造 `remote_im_followup` runtime context；委托续跑已显式构造 `delegate_continue` runtime context。

### 16.3 上下文整理 follow-up

`src-tauri/src/features/chat/model_runtime/tools_and_builtin/builtin_runtime_context.rs` 已在 `organize_context` 成功后补入内部 pending event：

1. `event_source=context_compaction`
2. `dispatch_reason=context_compaction_followup`
3. `activate_assistant=true`
4. 空 `messages`，只用于重新激活当前会话部门

如果当前会话已有 guided queue，则不额外插入 follow-up，避免重复激活。

### 16.4 前端统一草稿入口

`src/features/chat/composables/use-chat-flow.ts` 已新增 `beginAssistantActivationFromEvent(...)`：

1. 校验 `conversationId`
2. 读取 `activationId/requestId`
3. 新激活开始前清理当前会话旧流式缓存
4. 没有本地 queued round 时创建后台 queued round
5. 统一设置 waiting 状态并生成助理草稿

`handleExternalRoundStarted(...)` 已改为只通过该入口推进，不再依赖“用户发送路径已有 queued round”。

### 16.5 流式缓存轮次隔离

`conversationStreamCache` 已加入 `activationId` 字段。

清理覆盖：

1. 新 `round_started` 开始前
2. `round_completed`
3. `round_failed`
4. queued 阶段 completed / failed 终端事件
5. `CHAT_ABORTED_BY_USER`
6. queued stop
7. stop fallback
8. 新用户发送前空闲态

读取缓存时，如果当前存在 `activeActivationId`，缓存携带的 `activationId` 必须匹配，否则不恢复到草稿。

### 16.6 Delta 事件

`AssistantDeltaEvent` 已增加 `activationId` 字段；部分系统状态事件在 `core_send_inner.rs` 中使用当前 `runtime_context.request_id` 作为 activation 信息。底层模型流式片段仍有部分事件不携带 `activationId`，前端按当前 active round 保底接收；旧 activation 的 completed / failed 会被过滤。
