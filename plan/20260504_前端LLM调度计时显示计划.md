# 前端 LLM 调度计时显示计划

## 背景

当前聊天窗口中，助理流式响应期间消息头部会显示带流光效果的状态文案；流式完成后，该位置恢复为助理用户名与时间。用户希望在这一段“流光文案”中增加本地计时，并在最终消息同步完成后改用后端统计的真实调度耗时。

目标行为：

- 调度进行中：流光文案末尾显示本地计时，如 `正在思考中（1分23秒）`。
- 后端也记录整次 LLM 调度耗时，但流式开始前不要求前后端同步起点。
- 调度完成并同步正式助理消息后，前端显示以后端调度耗时为准。
- 完成后的耗时显示在气泡下方的悬停操作区，不再依赖是否执行工具。

## 当前链路事实

- 前端消息渲染在 `src/features/chat/components/ChatMessageItem.vue`。
- 流式头部文案由 `assistantStreamingHeaderStatus(block)` 生成，模板中以 `ecall-shimmer-text` 展示。
- 前端助理草稿由 `src/features/chat/composables/use-chat-flow.ts` 创建和更新，草稿 `providerMeta` 使用：
  - `_streaming`
  - `_preStreamingStatusText`
  - `_streamSegments`
  - `_streamAnimatedDelta`
  - `_streamToolCalls`
- 消息投影在 `src/features/chat/composables/use-chat-turns.ts`，会把草稿 meta 转成 `ChatMessageBlock`。
- 后端最终助理消息在 `src-tauri/src/features/system/commands/chat_and_runtime/core_send_inner.rs` 中持久化。
- 后端已有整轮调度起点 `chat_started_at`，并在 `chat_pipeline` 日志中使用它计算整体耗时。
- 工具聚合条在 `ChatMessageItem.vue` 中按 `toolCallsForBlock(block).length > 0` 渲染；最终耗时显示不再挂靠工具聚合条。

## 目标方案

### 1. 前端流式本地计时

在 `use-chat-flow.ts` 中为当前调度维护本地秒表：

- 调度开始时记录 `frontendDispatchStartedAtMs`。
- 每秒刷新一次 `frontendDispatchElapsedMs`。
- 将当前耗时写入助理草稿 `providerMeta`，例如：
  - `_frontendDispatchStartedAtMs`
  - `_frontendDispatchElapsedMs`
- 草稿更新、会话切换、stream rebind、队列恢复时，沿用现有 `conversationStreamCache` 机制保留该计时状态。
- 调度完成、失败、取消、history flushed 后清理 interval，避免后台残留计时器。

流式展示规则：

- `assistantStreamingHeaderStatus(block)` 在原有状态文案后拼接格式化耗时。
- 若无有效本地耗时，保持现有文案不变。
- 格式统一为中文紧凑形式：`X天X小时X分X秒`，低位至少显示秒。

示例：

- `正在等待回复（8秒）`
- `正在思考中（1分02秒）`
- `正在执行 shell_exec 中（2分15秒）`

### 2. 后端调度耗时落库

在 `core_send_inner.rs` 持久化最终助理消息前计算后端耗时：

- 使用已有 `chat_started_at.elapsed()` 作为整次调度耗时。
- 写入助理消息 `provider_meta`，建议字段名：
  - `dispatchElapsedMs`
- 字段只在最终正式消息上落库，不通过流式事件提前同步。
- 若本轮最终不生成助理消息（例如特殊工具结果抑制消息），不额外写入。

这满足“流式之前不需要同步，完成之后消息同步时覆盖为后端调度计时”的要求：前端草稿显示本地秒表，正式消息替换草稿后自然使用后端字段。

### 3. 前端类型与投影

更新前端类型与投影：

- `ChatMessage.providerMeta` 允许 `dispatchElapsedMs`。
- `ChatMessageBlock` 增加可选字段 `dispatchElapsedMs?: number`。
- `use-chat-turns.ts` 从 `providerMeta.dispatchElapsedMs` 读取后端耗时。
- 流式草稿也可从 `_frontendDispatchElapsedMs` 读取本地耗时，但该值只用于流式头部，不作为最终正式耗时。

### 4. 悬停操作区显示最终耗时

在 `ChatMessageItem.vue` 的完成态消息气泡下方操作区追加最终耗时：

- 只有非流式正式消息且 `dispatchElapsedMs > 0` 时显示后端耗时。
- 显示位置与复制、多选、隐藏气泡背景等操作同一行。
- 与底部操作按钮保持一致，仅在消息悬停时显示。

建议展示：

```text
用时 2分15秒  [多选] [复制] [隐藏气泡背景]
```

没有工具调用的普通回复也显示最终耗时。

### 5. 格式化函数

新增或就近实现一个纯函数：

```ts
formatDispatchElapsed(ms: number): string
```

规则：

- 小于 1 秒按 `0秒` 或 `1秒` 取整策略实现时确认，建议四舍五入后最小显示 `1秒`，避免完成瞬间显示空值。
- 天、小时、分钟为 0 时可省略高位。
- 秒始终显示。
- 示例：
  - `800` -> `1秒`
  - `65_000` -> `1分05秒`
  - `3_726_000` -> `1小时02分06秒`
  - `93_784_000` -> `1天02小时03分04秒`

## 实施步骤

### 阶段 1：计划确认后改前端草稿计时

1. 在 `use-chat-flow.ts` 增加本地计时状态与 interval 管理。
2. 在 `insertDraft` / `updateQueuedAssistantDraftStatus` / `updateDraftText` 中写入本地计时 meta。
3. 在成功、失败、取消、清屏、切换轮次等收尾路径清理计时器。
4. 在 `ChatMessageItem.vue` 读取流式本地耗时并拼到流光文案后。

### 阶段 2：后端最终耗时

1. 在 `core_send_inner.rs` 计算 `dispatchElapsedMs`。
2. 合并到最终助理消息 `provider_meta`。
3. 保持现有 `chat_pipeline` 日志耗时逻辑不变。

### 阶段 3：正式消息展示

1. 更新 `src/types/app.ts` 的消息类型。
2. 更新 `use-chat-turns.ts` 的 `ChatMessageBlock` 构造。
3. 在完成态消息气泡下方操作区追加 `用时 ...`。
4. 确保无工具消息同样显示最终耗时。

## 风险与注意事项

- 前端本地计时只用于流式体验，不作为最终真值；正式消息到达后以后端 `dispatchElapsedMs` 为准。
- 会话切换和 stream rebind 需要保留本地计时状态，否则回到会话后流光耗时会跳变或归零。
- 最终耗时显示在气泡下方的悬停操作区，并以正式消息里的后端 `dispatchElapsedMs` 为唯一显示条件。
- 收尾路径较多，计时器清理需要覆盖成功、失败、取消、队列无草稿完成、history flushed。
