# 20260417 models.dev 元数据缓存与配置页回填计划

## 目标

解决“模型元数据上限在配置窗口刷新后丢失”的问题：

- 后端缓存抓取到的 `models.dev api.json`
- 只有用户手动刷新模型时，才检查缓存是否超过 1 天并重新获取
- 前端每次切到 API 配置页时，向后端获取当前模型元数据
- 配置页据此临时回填：
  - `contextWindowTokens` 上限
  - `maxOutputTokens` 上限
  - `enableImage`
  - `enableTools`
  - `enableAudio`

明确边界：

- 不把具体模型上限写进 `app_config.toml`
- 不把具体模型上限写进 `app_data.json`
- 持久化的只有后端缓存的 `models.dev` 原始表数据

## 现状问题

- 当前配置页里的模型元数据上限主要存在前端局部状态 `modelCapabilityById`
- 该状态只在用户：
  - 选择模型
  - 输入模型后失焦
  - 回车确认模型
  时，通过 `fetch_model_metadata` 触发更新
- 整个配置窗口刷新后，前端局部状态会丢失
- 因此会出现：
  - 选择模型时上限正确
  - 刷新窗口后上限丢失

## 方案

### 1. 后端增加 models.dev 原始 JSON 缓存

新增后端缓存层，缓存抓取到的 `https://models.dev/api.json` 原始 JSON：

- 首次没有缓存：联网抓取并落地
- 平时页面读取：只读本地缓存，不触发联网更新
- 只有用户手动刷新模型时：
  - 有缓存且未超过 1 天：直接读本地缓存
  - 有缓存但超过 1 天：重新抓取并覆盖缓存
- 若手动刷新时重新抓取失败：
  - 有旧缓存则回退旧缓存
  - 无旧缓存则返回错误

缓存内容至少包含：

- 原始 JSON 文本或解析后的结构
- `updated_at`

### 2. 后端 fetch_model_metadata 改为只读缓存表

当前 `fetch_model_metadata` 是现拉现用，改为：

- 只通过本地缓存服务读取当前可用的 `models.dev` 数据
- 再在这份缓存数据里查找模型元数据

这样前端每次请求元数据时：

- 不需要直接联网
- 也不会因为窗口刷新而丢失能力上限来源
- 但返回的具体模型上限仍然只是当前页面的临时展示/约束结果，不写回配置

如果本地还没有缓存：

- 可返回“暂无模型元数据缓存，请先手动刷新模型”的结果或错误
- 不在页面读取链路里偷偷联网

### 3. 前端进入 API 配置页时主动回填当前模型元数据

在 API 配置页切入或当前 provider/model 变化时：

- 对当前 provider 下已有 `modelCard` 逐个调用 `syncModelMetadata`
- 用后端缓存表返回结果回填前端能力上限状态

这样即使：

- 重开窗口
- 切换配置页
- 切换当前 provider

也能重新拿回能力上限，而不是只在手动点选模型时正确

注意：

- 这里的“回填”仅指前端页面内存态
- 不做“把模型上限持久化到配置”的设计

## 实施步骤

1. 在 Rust 侧增加 `models.dev` JSON 缓存读写与过期判断
2. 改造 `fetch_model_metadata`，接入缓存表读取逻辑
3. 在“手动刷新模型”链路中加入 `models.dev` 缓存过期检查与按需重新抓取
4. 在前端 `ApiTab.vue` 中增加“进入页面/切换模型卡时自动回填元数据”的触发
5. 保持现有“手动选择模型时同步元数据”的行为不变
6. 不改 `AppConfig` / `app_data` 结构，不新增“模型上限持久化字段”

## 预期结果

- 刷新整个配置窗口后，模型能力上限不会再丢失
- 配置页切回来时，当前模型的能力上限可以自动恢复
- 正常打开配置页不会频繁联网请求 `models.dev`
- 只有用户手动刷新模型时，才会触发 `models.dev` 过期检查与按需更新
