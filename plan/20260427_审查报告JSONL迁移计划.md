# 20260427_审查报告 JSONL 迁移计划

## 当前状态

本计划已经从“待迁移方案”变成“实现回顾文档”。旧模型中“报告伪装成聊天消息、挂回 batch.report、前端围绕 currentBatch.report 展示”的链路已退出主路径。

当前真实模型：

1. 审查报告以每会话独立 JSONL 报告流存储。
2. 生成审查报告时先创建 `pending` 草稿记录，再更新为 `success` 或 `failed`。
3. 侧边栏已拆成“工具解释 / 审查报告”两个 tab。
4. 报告列表直接读取报告记录，不再依赖当前批次的 `report` 槽位。
5. 成功报告的主数据是副手最终原始文本 `reportText`；前端解析 JSON 成功才渲染结构化 UI，解析失败展示原文。
6. 旧 `tool_review_report` 消息只作为启动迁移输入，不再作为长期读取来源。

仓库里已有可复用的 JSONL 存储基建：

- `src-tauri/src/features/chat/message_store/paths.rs` 提供按 `conversation_id` 校验路径与原子写工具。
- `src-tauri/src/features/chat/message_store/jsonl_snapshot.rs` 提供 JSONL 编码/解码思路。
- `src-tauri/src/features/chat/message_store/persist.rs` 已有 append / replace / splice 的原子落盘套路。

这次不做长期兼容保留，旧消息只用于一次性迁移：写新数据 → 读取验证 → 删除旧数据。

另一个关键前提也已确认：项目启动期本来就有专门迁移链路，消息 JSONL 迁移已经采用 `plan / write / verify / cleanup-old` 这类模式。因此，旧 `tool_review_report` 消息清理不该散落在业务提交链路里，而应挂到启动迁移模块处理。

## 目标

把工具审查报告从“会话消息伪装存储”迁移为“每会话独立 JSONL 报告流”，并同步改造侧边栏展示。

落地结果：

1. 每个会话单独一份审查报告 JSONL。
2. 每次生成审查报告都先创建一条草稿记录。
3. 报告状态仅三种：`pending`（还未生成）、`failed`（生成失败）、`success`（生成成功）。
4. 每条记录至少包含：审查范围、工作目录、审查结果。
5. 工具审查侧边栏拆成两个总 tab：`工具解释`、`审查报告`。
6. 审查报告 tab 按新到旧列出当前会话的报告记录，显示状态，点击后查看详情。
7. 旧 `tool_review_report` 消息链路直接移除，不保留兼容读写。

## 非目标

- 不保留旧消息报告兼容层。
- 不保留旧 `tool_review_report` 消息作为长期兼容读取源。
- 不把副手结果转成 Markdown 后再保存。
- 不改动 commit 分页、多选批次、多选 commit 这套已确定的目标选择交互。
- 不新增主链路 git 分析逻辑；除 commit 列表外，审查材料仍由子代理在工作区内自行获取。

## 新存储模型

### 文件布局

每个会话单独一份报告文件，建议放在会话目录旁的独立分片目录，例如：

`<data>/tool-review-reports/<conversation_id>/reports.jsonl`

要求：

1. `conversation_id` 必须做和 message store 同等级别的合法性校验，禁止路径穿越。
2. 写文件必须使用原子替换或受控 append，不能直接裸写。
3. 路径解析逻辑单独封装，避免和 message store 共用会话消息语义。

### 记录结构

每行一条 JSON 记录，建议字段如下：

```json
{
  "id": "uuid",
  "conversationId": "conv_xxx",
  "createdAt": "2026-04-27T05:20:18+08:00",
  "updatedAt": "2026-04-27T05:20:18+08:00",
  "status": "pending",
  "scope": "batch | commit | main | uncommitted | custom",
  "target": "原始目标字符串",
  "workspacePath": "E:/github/easy_call_ai",
  "reportText": "副手最终原始文本，通常是 JSON 字符串，也可能是解析失败时的原文",
  "errorText": "失败原因",
  "batchKeys": ["msg_xxx"],
  "commitHashes": ["abc123"],
  "customTargetText": "审查目标说明"
}
```

字段口径：

- `status`：前端唯一状态源。
- `scope`：对应当前五类审查目标。
- `target`：保留前端提交给后端的原始目标文本，便于回显与排障。
- `workspacePath`：记录实际使用的主工作区路径。
- `reportText`：成功时保存副手最终原始文本，不做 Markdown 转换。
- `errorText`：失败时保存失败原因。
- `batchKeys` / `commitHashes` / `customTargetText`：用于详情回显，不再塞进聊天消息 meta。

## 后端迁移方案

### 0. 启动期迁移挂载点

这次迁移要接到启动期迁移链，而不是等用户点开侧边栏后再临时处理。

已确认的现有事实：

1. [app_data_layout.rs](E:/github/easy_call_ai/src-tauri/src/features/config/app_data_layout.rs) 的 `read_app_data(...)` 已承担启动期兼容迁移职责，会在读取时做旧结构修正并在需要时回写。
2. [message_store/migration.rs](E:/github/easy_call_ai/src-tauri/src/features/chat/message_store/migration.rs) 已有成熟迁移模式：先规划、写入新存储、校验可读性，再决定旧状态如何处理。
3. 运行态读取入口 [runtime_cache.rs](E:/github/easy_call_ai/src-tauri/src/features/core/domain/runtime_cache.rs) 会通过 `read_app_data(&state.data_path)` 汇入启动后的主读取链。

因此，这次工具审查报告迁移建议挂在 `app_data` 读取后的启动迁移流程里：

1. 读取当前会话聚合数据。
2. 扫描每个 conversation 里的旧 `tool_review_report` 消息。
3. 将这些旧报告写入对应的 per-conversation report JSONL。
4. 重新读取并校验新 JSONL 内容是否完整。
5. 仅在校验通过后，从 `conversation.messages` 删除旧 `tool_review_report` 消息并回写会话。
6. 任何一步失败，都保留旧消息，不删旧数据。

这和你给的口径一致：`写入新数据 → 读取验证 → 删除旧数据`。

### 1. 新增 tool review report store

新增独立报告存储模块，职责只做四件事：

1. 解析报告文件路径。
2. 读取当前会话全部报告记录。
3. 追加一条草稿记录。
4. 按 `id` 更新一条已有记录状态与内容。

实现要求：

- 复用 message store 的原子写风格。
- 允许“读取全量 -> 替换单条 -> 原子重写”这条最小实现，不为这次需求过早上索引。
- 读列表时直接按文件顺序读取，返回给前端前再转成新到旧。

### 2. 改造 submit_tool_review_code / submit_tool_review_batch

无论是代码审查还是批次报告，提交流程都改成：

1. 先解析当前会话主工作区路径。
2. 立即创建一条 `pending` 报告记录。
3. 执行原本的报告生成逻辑。
4. 成功则把该记录更新成 `success`，写入 `reportText`。
5. 失败则把该记录更新成 `failed`，写入 `errorText`。
6. 返回 `reportId` 与最新记录，不再返回 `reportMessageId`。

注意：

- “预先生成草稿”必须发生在实际报告生成之前。
- 即使生成失败，前端列表里也必须能看到这条失败记录。

### 3. 旧消息链路状态

本次要区分两件事：

1. **运行时写链路**：立刻停止再写旧消息。
2. **历史数据清理**：交给启动迁移模块，在“新写入成功且校验通过”后删除旧消息。

运行时主链路已不再写入旧消息；以下旧逻辑仅可作为迁移输入或待清理残留看待，不能再作为报告展示真相源：

1. `tool_review_report_from_message(...)`：仅用于旧消息迁移读取。
2. `tool_review_upsert_report_message(...)`：不应再出现在运行时写链路。
3. `collect_tool_review_batches_internal(...)` 中报告消息扫描与 `batch.report` 回填：不应再作为前端报告来源。
4. `reportMessageId` 返回值与基于消息 ID 的刷新逻辑：不应再作为新报告链路依赖。

历史存量数据层面：

- 旧 `tool_review_report` 消息不再作为长期兼容读取来源。
- 首次启动迁移成功后，再从旧消息中清走。
- 若迁移失败，则旧消息暂留，避免丢档。

结果是：

- `ToolReviewBatchSummary` 不再携带 `report?: ToolReviewReportSummary`。
- 工具批次只负责工具解释，不再混入历史报告。

### 4. 新增后端接口

至少补四个接口：

1. `list_tool_review_reports(conversationId)`
   - 返回当前会话全部报告，顺序为新到旧。
2. `get_tool_review_report_detail(conversationId, reportId)`
   - 返回单条报告详情。
3. `submit_tool_review_batch(...)`
   - 返回 `reportId` + `report`，不再返回消息 ID。
4. `submit_tool_review_code(...)`
   - 返回 `reportId` + `report`，不再返回消息 ID。

若前端只需要列表即可打开详情，也可以先不单独做 detail 接口，直接让列表返回完整详情字段；但若要减轻列表负载，则分列表摘要与详情接口。实现时按最小改动裁决。

- `reportMessageId` 刷新逻辑

### 3. 生成后的前端行为

用户点击“生成审查报告”后：

1. 后端先落一条 `pending`。
2. 前端刷新报告列表，立即能看到“还未生成”。
3. 请求结束后刷新列表。
4. 成功时状态变成“生成成功”，失败时状态变成“生成失败”。
5. 用户点击该记录后查看详情。详情页以 `reportText` 为主数据：JSON 解析成功渲染 UI，失败显示原文。

报告查看入口是报告 tab 中的记录点击，不再从当前批次直接打开 `currentBatch.report`。

## 风险点

1. 若前端重新读取 `currentBatch.report`，会回到旧模型并造成报告入口错位。
2. 若后端重新依赖 `reportMessageId`，会破坏报告 JSONL 主链路。
3. 侧边栏最近刚因模板变量误用发生过挂载崩溃，这次双 tab 改造后必须先查模板引用是否都基于现有状态源。
4. 报告 JSONL 是新增分片，路径与会话 ID 校验必须先做好，不能裸拼字符串。
5. 启动迁移如果删旧过早，会直接损坏历史存档，所以必须严格遵守“先写新、再读验、后删旧”。

## 已确认决策

1. 旧实现直接迁移，不保留兼容层。
2. 计划文件放在 `plan/` 目录。
3. 审查报告按“每会话单独 JSONL + 每次生成新增一条记录”建模。
4. 状态只分为：还未生成、生成失败、生成成功。
5. 侧边栏拆成“工具解释 / 审查报告”两个 tab。
6. 历史旧报告迁移挂到启动期迁移模块，执行顺序为：写新数据 → 读取验证 → 删除旧数据。
