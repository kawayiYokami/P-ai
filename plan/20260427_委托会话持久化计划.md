# 20260427_委托会话持久化计划

## 当前状态

本计划已进入实现后回顾状态。最初触发点是代码审查委托运行期出现 `Conversation 'delegate-...' not found.`，根因是委托 runtime thread 与持久化读取链混读。

当前真实实现已经收口为：

1. 委托会话独立持久化到 `<app_root>/delegate-conversations/`。
2. 委托会话强制 `conversation_kind = delegate`。
3. 委托会话必须携带 `root_conversation_id`，作为所属主会话 ID 真相源。
4. 委托线程创建时立即写入持久化文件；消息与 Todo 更新通过统一入口回写。
5. 归档窗口“委托会话”列表合并 active runtime、recent runtime 与持久化文件，持久化数据不再依赖内存缓存存活。
6. 委托会话读取入口支持 active / recent / persisted 三层查找，避免再次掉回普通主会话仓库导致 `delegate-... not found`。

这份文档保留为后续回顾委托会话架构时的事实说明，不再作为待实施计划使用。

## 已落地目标

委托会话已经被定义为一类独立持久化对象，单独存储，统一以 `root_conversation_id` 表达所属主会话，并让发送链、归档链、恢复链读取同一套委托会话数据。

落地结果：

1. 每个委托会话都有独立持久化文件。
2. 委托会话记录中明确保存 `root_conversation_id`。
3. 委托创建后立即落盘，后续消息追加与状态变化持续回写。
4. `delegate-...` 会话读取不再半路回落到普通 conversation 仓库。
5. 归档窗口“委托会话”tab 以持久化委托会话为主，runtime/recent 补充热态信息。
6. 代码审查这类上游业务再次拉起委托时，不再因 `delegate-... not found` 失败。

## 非目标 / 已明确不做

1. 不重做委托工具的上游语义；本次不改“委托工具强制以副手部门类型启动”的规则。
2. 不重做委托调度业务分层；本次只修正会话持久化与读取一致性。
3. 不强制为历史残缺委托记录伪造消息正文。
4. 不把委托会话直接并入普通主会话 shard，避免污染前台主会话列表。
5. 不顺手重构无关的模型选择、审批、工具协议。

## 新语义定义

### 1. 委托会话是什么

委托会话是由委托调度拉起、拥有独立 `conversation_id` 的子会话对象。它不是普通主会话的运行时幻影，也不是仅供 UI 展示的线程快照，而是一类需要持久化的正式会话记录。

### 2. 所属主会话如何表达

统一使用 `root_conversation_id` 作为所属主会话 ID 真相源。

约束如下：

1. 委托会话创建时必须写入 `root_conversation_id`。
2. 若委托来自主会话，则 `root_conversation_id = 主会话 conversation_id`。
3. 若未来存在委托嵌套，仍以最外层主会话 ID 作为 `root_conversation_id`，避免一层层漂移。

### 3. runtime thread 的地位

委托 runtime thread 继续保留，但降级为“热运行缓存”，不再承担唯一真相源职责。

也就是说：

1. 运行时内存线程负责执行态调度与热更新。
2. 持久化委托会话负责恢复、归档、统一读取。
3. 二者冲突时，以持久化会话结构为基础，runtime 只补充最新热状态。

## 已落地存储方案

### 1. 文件布局

在现有主会话存储体系之外，为委托会话建立独立目录：

`<data>/delegate-conversations/`

每个委托会话单独一个 JSON 文件，文件名使用委托会话 ID。

要求：

1. 路径解析与 ID 校验独立封装，避免路径穿越。
2. 读写采用现有存储层已经在用的原子替换/受控写入模式。
3. 委托会话目录与普通 conversation shard 分离，避免主会话列表误收录。

### 2. 持久化内容

委托会话至少包含：

1. `conversation_id`
2. `conversation_kind = delegate`
3. `root_conversation_id`
4. 当前消息列表或消息引用
5. `delegate_id`
6. 创建时间、更新时间、归档时间
7. 创建时间、更新时间

原则：优先复用现有 conversation 结构中已稳定的字段，不为这次需求额外发明平行概念。

## 已落地读取链

### 1. 创建链路

`delegate_runtime_thread_create(...)` 不再只写 runtime 线程表；创建委托线程时同步完成：

1. 构造委托会话对象。
2. 写入 `delegate-conversations` 目录。
3. 将该持久化对象注册到 runtime thread 对应的热缓存。

这样一来，委托会话从出生起就有可恢复真相源。

### 2. 更新链路

委托运行期间，以下动作会同步回写持久化委托会话：

1. 用户侧/系统侧消息追加。
2. assistant delta 收束成正式消息。
3. Todo 更新。
4. 完成/失败时写入 `archived_at` 并进入 recent runtime。

目标很直：不再允许“内存线程里有，持久化里没有”的长期分叉。

### 3. 发送链路

`send_chat_message_inner` 及其提示词准备链遇到委托会话时，统一走“委托会话读取入口”，不再中途掉回普通 conversation 读取。

具体要求：

1. 先按委托会话 ID 从 `delegate-conversations` 读取。
2. 若 runtime thread 存在，则以其热态覆盖部分运行时字段。
3. 若 runtime thread 不存在，但持久化文件存在，仍可正常读取并继续恢复/展示。
4. 普通 `state_read_conversation_cached(...)` 之类主会话读取路径，不再直接承担 `delegate-...` 会话解析。

这一步是消除 `Conversation 'delegate-...' not found` 的核心。

## 归档与展示现状

归档窗口“委托会话”tab 已改为以持久化委托会话列表为主数据源。

当前口径：

1. 先读取 `delegate-conversations` 列表。
2. 再用 active runtime / recent runtime 补充热状态字段。
3. 展示层按委托会话自身更新时间排序。

结果是：

1. 已完成的委托会话在重启后仍可见。
2. 当前运行中的委托会话仍可展示实时状态。
3. UI 不再依赖“最近线程缓存还没丢”这种偶然性。

## 历史兼容策略

对历史数据分两类处理：

### 1. 新创建委托会话

从本次改造落地后开始，新委托会话一律完整持久化。

### 2. 旧历史委托记录

若历史数据只有 delegate record、runtime snapshot 或 recent thread 信息，但缺少完整正文，则按“历史残缺记录”处理。

处理原则：

1. 可以展示已有元信息。
2. 不伪造不存在的消息正文。
3. 不为了兼容而继续维持 runtime-only 作为正式真相源。

若未来补历史迁移，前提必须是有真实可验证原始数据，不能猜写。

## 实际影响范围

### 后端

重点会落在以下几类模块：

1. 委托会话/委托记录存储层
2. 委托 runtime thread 创建与更新链
3. `send_chat_message_inner` 与提示词准备链
4. 归档窗口委托会话列表接口

### 前端

前端跟随调整委托会话列表来源与展示字段；普通主会话列表不应收录委托会话。

## 风险与控制

### 风险 1：主会话与委托会话再次混仓

控制：物理目录分离，读取入口分离，列表展示分离。

### 风险 2：runtime 与持久化双写不一致

控制：以持久化委托会话为基础真相源，runtime 仅做热缓存；所有状态变更都沿同一写入口回写。

### 风险 4：前端 presence 把历史委托误判为运行中

控制：当前已临时隐藏工具栏人格头像后台任务红点。后续如恢复该红点，不能再用全量委托会话列表反推运行态，应改用真实 active runtime 或明确的委托运行状态。

### 风险 3：历史委托记录不完整

控制：只迁移有真实数据支撑的部分；缺失正文的旧记录按残缺历史处理，不伪造。

## 验收标准

满足以下条件，才算这次改造成立：

1. 新委托会话创建后，磁盘上能看到独立持久化文件。
2. 该文件中能读到 `root_conversation_id`，且值等于所属主会话 ID。
3. 委托运行期间，消息与状态变化能持续回写。
4. 应用重启后，委托会话仍可从持久化目录恢复并在归档页可见。
5. 代码审查链再次拉起委托时，不再出现 `Conversation 'delegate-...' not found.`
