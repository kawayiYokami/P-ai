# 20260422_请求日志按调度聚合展示计划

## 背景

当前设置页“日志”标签展示的是最近 10 条 `LlmRoundLogEntry`。但现有埋点粒度同时包含：

- `chat`：单次模型轮次
- `chat_pipeline`：整次调度管线

当一次调度内部发生多次工具调用时，会形成“多轮 `chat` + 一条 `chat_pipeline`”的组合。结果就是：

- 同一次调度会在列表中占据多条卡片
- `chat_pipeline` 与 `chat` 的视觉层级几乎一致
- 用户看到的是“碎片化轮次日志”，而不是“一次调度的整体过程”
- 最近 10 条容量会被工具循环很快挤满，失去“最近 10 次请求”的语义

## 目标

本次改造要达成以下目标：

1. 日志页主列表从“最近 10 条轮次日志”改为“最近 10 条调度日志”。
2. 单次调度内部的多轮 `chat` 不再平铺占坑，而是聚合到所属 `chat_pipeline` 下。
3. `chat_pipeline` 的展示层级要明显区别于单轮 `chat`。
4. 页面展示避免深层嵌套，优先提供“调度概览 -> 轮次摘要 -> 详情面板”的阅读路径。

## 非目标

本次不做以下内容：

- 不改运行时后台日志（`runtime_log_*`）页面
- 不调整工具调用本身的协议与执行流程
- 不修改 `chat` / `chat_pipeline` 的实际推理逻辑
- 不把现有请求日志持久化到磁盘，仍保持内存态最近记录

## 当前实现结论

### 数据记录层

- `push_llm_round_log(...)` 会向 `llm_round_logs` 追加一条 `LlmRoundLogEntry`
- `router.rs` 在每次模型轮次结束后写入一条 `scene = "chat"` 日志
- `core_send_inner.rs` 在整次聊天调度结束后写入一条 `scene = "chat_pipeline"` 日志
- 当前容量 `LLM_ROUND_LOG_CAPACITY = 10`

### 展示层

- `LogTab.vue` 直接把 `list_recent_llm_round_logs()` 返回结果倒序平铺
- 每条 entry 统一用同一种卡片展示
- 页面没有“按 traceId 聚合”“按调度分组”“按轮次分层”的概念

## 目标方案

### 一、数据模型收口为“每次调度一条主日志”

将最近日志的主存储单位从 `LlmRoundLogEntry` 调整为“聚合后的 pipeline 日志”。

建议新增聚合视图模型（命名可实现时再定）：

- `LlmPipelineLogEntry`
  - `id`
  - `createdAt`
  - `traceId`
  - `scene = "chat_pipeline"`
  - `provider`
  - `model`
  - `requestFormat`
  - `baseUrl`
  - `elapsedMs`
  - `success`
  - `error`
  - `timeline`
  - `request`
  - `response`
  - `headers`
  - `tools`
  - `roundCount`
  - `toolCallCount`
  - `rounds: LlmRoundLogEntry[]`

其中：

- `rounds` 保存原本同一次调度里的各轮 `chat`
- 主列表只保留 `chat_pipeline`
- 容量改为“最近 10 条 pipeline”

### 二、聚合策略

按 `traceId` 聚合同一次调度内的日志。

建议规则：

1. `scene = "chat"` 的 entry 不直接进入主列表，而是先进入“待归并轮次缓存”。
2. 当同 `traceId` 的 `chat_pipeline` 到来时：
   - 取出之前缓存的全部 `chat`
   - 组装为一条完整 `LlmPipelineLogEntry`
   - 推入最近日志队列
3. 如果调度失败但仍有 `chat_pipeline`，同样按失败 pipeline 收口。
4. 如果极端情况下存在孤立 `chat` 且迟迟没有 `chat_pipeline`：
   - 暂定不单独显示在主列表
   - 可在实现中增加超时兜底策略，但不是第一阶段必要项

### 三、展示方案

避免深层嵌套，采用“三段式阅读路径”：

1. 调度主列表
   - 一张卡代表一次调度
   - 显示摘要：
     - 时间
     - provider / model
     - 成功 / 失败
     - 总耗时
     - 总轮次
     - 总工具次数

2. 调度展开区
   - 不再平铺 `request/response/tools` 多层折叠
   - 改为优先显示：
     - 调度概览
     - timeline
     - 轮次摘要列表

3. 轮次详情面板
   - 点击某一轮后，在抽屉或弹窗中展示详情
   - 详情采用 tabs：
     - `Request`
     - `Response`
     - `Tools`
     - `Headers`
     - `Error`

### 四、视觉层级建议

`chat_pipeline` 与 `chat` 明显区分：

- `chat_pipeline`
  - 作为主卡片
  - 更强标题与摘要
  - 更明显的状态标识

- `chat`
  - 不再以同级卡片平铺
  - 仅作为“轮次列表中的子项”
  - 样式更轻、更紧凑

## 具体改造步骤

### 阶段 1：后端日志聚合

1. 梳理 `push_llm_round_log(...)` 的调用点
2. 新增“按 traceId 暂存 chat 轮次”的内存结构
3. `chat` entry 先入暂存，不直接进入最近日志主队列
4. `chat_pipeline` 到来时，将同 traceId 的 `chat rounds` 聚合成单条 pipeline 记录
5. 将容量规则改为“最近 10 条 pipeline”
6. 清空日志时，同时清理 pipeline 队列与中间暂存

### 阶段 2：前端类型与 API 对齐

1. 新增前端 pipeline 日志类型
2. 调整 `LogTab.vue` 读取的数据结构
3. 保持“刷新 / 清空”交互不变

### 阶段 3：日志页 UI 重做

1. 主列表改为调度卡片
2. 卡片头部展示调度摘要
3. 展开区显示 timeline 与轮次列表
4. 新增轮次详情抽屉或弹窗
5. 将原本平铺的 `Headers / Tools / Request / Response` 重排到详情视图

## 风险与注意事项

### 1. traceId 归并稳定性

若个别场景 `chat` 与 `chat_pipeline` 的 `traceId` 不一致，会导致无法归并。实现前需确认：

- 常规聊天
- fallback
- 失败
- 工具循环

这些路径上的 traceId 是否稳定一致。

### 2. 中途失败与孤立 chat

若只产生了 `chat`，但没有最终 `chat_pipeline`，会出现“轮次已记录但主日志未生成”的情况。需要决定：

- 是丢弃
- 是超时兜底生成失败 pipeline
- 还是保留临时缓存直到清空

第一阶段建议先以“正常完成/失败 pipeline 收口”为主，不额外复杂化。

### 3. 旧 UI 信息密度下降

聚合后主列表更清爽，但首次打开时看到的信息会更少。需要通过：

- 摘要字段
- 轮次列表
- 详情抽屉

来补足诊断信息，避免“信息被藏太深”。

## 验证方案

实现后至少验证以下场景：

1. 无工具调用
   - 一次调度仅显示 1 条 pipeline
   - 轮次为 1

2. 多次工具调用
   - 7 次工具 + 完成，只显示 1 条 pipeline
   - `roundCount` 与内部实际 chat 轮数一致

3. 请求失败
   - 主列表仍保留 1 条失败 pipeline
   - 错误摘要正确显示

4. 清空日志
   - 主列表清空
   - 中间缓存也一并清空

5. 最近 10 条容量
   - 以 pipeline 为单位裁剪，不再被单次多轮调度快速挤爆

## 预期结果

改造完成后，日志页会从“最近 10 条碎片轮次日志”变成“最近 10 次调度日志”：

- 一次调度只占 1 张主卡
- 工具循环不再造成 8 张平级卡片
- `chat_pipeline` 成为真正的主视图
- `chat` 成为该调度内部的轮次细节
- 页面更贴近“排障一次请求”而不是“阅读一串原始埋点”
