# 20260423_聊天大消息渲染性能优化计划

## 一、背景

当前聊天前端在加载“庞大的消息”时，已经出现明显不可接受的卡顿。

这次排查先不直接进入实现，而是先把真实瓶颈、方案取舍和推荐施工顺序写清楚，避免后续一边改一边重新判断方向。

本次重点回答两个问题：

1. 当前卡顿到底是不是“消息条数太多”导致。
2. `倒置滚动区域` 与 `虚拟滚动` 两个思路，哪个更值得优先落地。

---

## 二、已确认事实

### 1. 当前前台并不是把整段历史全量挂到聊天窗口

前台加载消息时，实际只保留最近 `50` 条：

- `src/features/chat/composables/use-chat-runtime.ts:177-190`

也就是说，当前“加载很卡”并不等价于“聊天窗口里塞了几百上千条消息”。

更准确的说法是：

- **即使只显示最近 50 条，只要其中有若干条超长 Markdown / 代码块 / Mermaid / 多媒体消息，前端依然会卡。**

### 2. 当前消息列表仍然是“全量真实渲染”

`ChatView.vue` 中直接对 `chatRenderItems` 做 `v-for`，每个 item 都会进入真实组件树：

- `src/features/chat/views/ChatView.vue:31-170`

虽然这里已经用了 `v-memo`：

- `src/features/chat/views/ChatView.vue:106`
- `src/features/chat/views/ChatView.vue:146`

但 `v-memo` 只能减少“后续不必要更新”，不能降低“初次挂载全部 DOM”的成本。

### 3. 单条 assistant 消息本身非常重

`ChatMessageItem.vue` 中 assistant 消息默认直接上 `MarkdownRender`：

- meme 文本段：`src/features/chat/components/ChatMessageItem.vue:231-246`
- 主正文：`src/features/chat/components/ChatMessageItem.vue:267-283`

而且当前策略是：

- 流式中才启用 batch rendering
- 一旦消息不是 streaming，就改成一次性完整渲染

对应代码：

- `src/features/chat/components/ChatMessageItem.vue:272-278`

这意味着：

- **一条已经落盘的超长消息，在首次显示时会尝试在很短时间内完成整段 Markdown AST -> 组件树 -> DOM 挂载。**

### 4. 单条消息渲染后还会做 DOM 扫描

每条消息渲染完成后，会在 `watchPostEffect` 里执行：

- `src/features/chat/components/ChatMessageItem.vue:799-809`
- `src/features/chat/components/ChatMessageItem.vue:930-933`

这里会：

1. `nextTick`
2. `querySelectorAll("a[href]")`
3. 遍历所有链接并重写本地路径

如果一条大消息里链接很多，或者 Markdown 节点很多，这又会增加一次渲染后的主线程工作。

### 5. 消息投影和签名阶段也有大文本 CPU 开销

`use-chat-turns.ts` 会对消息做签名和投影：

- `src/features/chat/composables/use-chat-turns.ts:23-35`
- `src/features/chat/composables/use-chat-turns.ts:56-63`
- `src/features/chat/composables/use-chat-turns.ts:153-166`

当前签名里直接包含：

- `JSON.stringify(message.parts || [])`
- `JSON.stringify(message.extraTextBlocks || [])`
- `JSON.stringify(message.providerMeta || {})`
- `JSON.stringify(message.toolCall || [])`

这对“大文本 + 大工具参数 + 大附件元数据”消息并不便宜。

### 6. 当前滚动与对齐逻辑强依赖“正常方向滚动”

聊天滚动当前明确按“顶部 0，底部 scrollHeight”来处理：

- `src/features/chat/composables/use-chat-scroll-layout.ts:41-48`
- `src/features/chat/composables/use-chat-scroll-layout.ts:68-86`
- `src/features/chat/composables/use-chat-scroll-layout.ts:101-113`

同时还叠加了：

- `sticky top-0` 的 Todo 区
- active turn 顶部对齐
- jump-to-bottom 按钮

对应模板位于：

- `src/features/chat/views/ChatView.vue:31-85`

这说明如果改成倒置滚动，不是只改个 CSS 那么简单，而是会牵动现有整套滚动语义。

---

## 三、问题定义

当前卡顿本质上是三个问题叠加：

1. **全量挂载问题**
   当前可见列表中的每一条消息都是真实组件树，重消息越多，初次挂载越慢。

2. **单条重消息问题**
   超长 Markdown / 代码块 / Mermaid / 多图消息在非 streaming 状态下会一次性完整渲染，单条就可能把主线程卡住。

3. **派生计算问题**
   消息签名、显示投影、渲染后 DOM 扫描等操作，会在“消息很大”时继续放大成本。

所以这里不能只用“消息列表优化”来描述问题。  
更准确的目标应该是：

- **既减少同时存在的重 DOM 数量，也降低单条重消息首帧渲染成本。**

---

## 四、对两个思路的判断

### 1. 倒置滚动区域

#### 适合解决什么

倒置滚动更适合解决这些问题：

- 聊天最新消息天然贴近视觉底部
- 向上加载历史时更容易维持视觉锚点
- 某些“prepend 老消息”场景下滚动跳动更少

#### 不能直接解决什么

它**不能直接减少**：

- DOM 节点数量
- Markdown 解析成本
- 代码块 / Mermaid 组件挂载成本
- 单条大消息渲染后的布局与绘制成本

也就是说：

- **倒置滚动不是本次卡顿的主解。**

#### 当前项目里的额外代价

由于当前已有：

- `scrollToBottom`
- `isNearBottom`
- active turn 顶部对齐
- sticky Todo
- jump-to-bottom

一旦改成倒置滚动，几乎整套滚动语义都要重写。

结论：

- **倒置滚动可以作为后续“向上增量加载历史”的可选增强，但不建议作为本次性能治理的第一优先级。**

### 2. 虚拟滚动

#### 能解决什么

虚拟滚动是真正能降低以下成本的手段：

- 同时挂载的消息组件数量
- 同时存在的 Markdown DOM 数量
- 滚动时浏览器的布局 / 绘制压力

#### 当前需要注意的现实约束

聊天消息不是固定高度列表，而是：

- 文本高度差异极大
- 代码块高度不固定
- Mermaid 高度不固定
- 图片和附件也会改变高度
- 还存在 `group` / `message` / `compaction` 三种 render item

所以不能直接照搬日志弹窗那种固定行高虚拟列表。

项目里已有固定高度虚拟列表参考：

- `src/features/shell/components/RuntimeLogsDialog.vue`

但聊天区需要的是：

- **动态高度虚拟滚动**
- **高度测量与缓存**
- **滚动锚点保持**
- **流式中的尾部 pin 住**

#### 还要再补一句

如果只有 1 到 3 条消息，但每条都极其巨大，那么“只做消息级虚拟滚动”也不够。

因为：

- 这些重消息本身仍会在进入 viewport 时一次性完整渲染

所以本次不应把虚拟滚动理解成唯一解，而应理解成：

- **主解之一**
- **必须配合单条重消息降载策略一起上**

结论：

- **虚拟滚动是正确方向，但要做成“动态高度 + 尾部保活 + 重消息延迟激活”的版本，不能只做最简化名单式 windowing。**

## 五、推荐方案

推荐采用“两阶段主改造 + 一项可选增强”的顺序。

### 第一阶段：先压单条重消息成本

这一阶段目标不是彻底解决所有问题，而是先把最明显的主线程尖峰降下来。

建议优先做：

1. 大消息在非 streaming 状态下也支持分批渲染
   当前只有 streaming 才 batch render，落盘后的大消息仍一次性渲染，建议给超长正文开启受控 batch。

2. 视口外消息延迟激活重内容
   先渲染轻骨架或简化占位，进入 viewport 后再挂 `MarkdownRender`、Mermaid、重代码块。

3. 去掉每次渲染后的全量链接扫描
   尽量改成“渲染前标准化 href”或“仅首次 final 后处理一次”，不要让每次更新都 `querySelectorAll("a[href]")`。

4. 减少重复的大字符串签名
   `messageSignature()` 与消息等价比较里对大对象 `JSON.stringify` 的次数要降下来，最好改成增量签名或更轻的内容版本号策略。

这一阶段的意义是：

- 即使虚拟滚动还没上，单条超长消息首次显示也不会那么炸。

### 第二阶段：引入聊天区动态高度虚拟滚动

建议在 `ChatView.vue` 这一层按 `ChatRenderItem` 做虚拟化，而不是在 `ChatMessageItem` 内部零散做隐藏。

推荐原则：

1. 虚拟化粒度使用当前已有的 `chatRenderItems`
   这样可保留 `message` / `group` / `compaction` 现有语义。

2. 维护高度缓存
   初次估高，挂载后测量回写，高度变化时修正滚动偏移。

3. 尾部保活
   靠近当前活跃轮次、正在 streaming 的消息，以及 jump-to-bottom 附近区域要始终挂载，避免流式中闪烁。

4. 合理 overscan
   防止滚动过快时白屏，但不要让 overscan 大到失去意义。

5. 与当前滚动语义兼容
   保留现有正常方向滚动，不要和倒置滚动一起上，避免同一阶段复杂度失控。

这一阶段完成后，才会真正减少“同屏和近屏内挂着太多重消息”的问题。

### 第三阶段：倒置滚动仅作为可选增强

只有在后续确认要做这些能力时，再考虑倒置滚动：

- 真正的向上无限历史加载
- prepend 历史时必须极稳的视觉锚点
- 想让聊天列表的底部语义更自然

如果没有这些明确收益，本阶段可以不做。

---

## 六、实施建议

### 1. 先不要把“倒置滚动 + 虚拟滚动”捆绑成一个需求

这两个改造都不小，而且目标不同：

- 虚拟滚动解决性能
- 倒置滚动更多解决聊天滚动体验和历史加载语义

把它们绑在一起，只会增加调试面。

### 2. 先做低风险基线优化，再做虚拟滚动

原因很简单：

- 如果连单条重消息都没降载，虚拟滚动上线后仍可能在“滚到某条超长消息进入视口”时卡一下

所以第一阶段不是可有可无，而是虚拟滚动前的铺路。

### 3. 虚拟滚动应尽量复用现有状态语义

当前已有：

- `chatRenderItems`
- `activeTurnGroupId`
- `activeTurnUserId`
- `use-chat-scroll-layout`

建议优先围绕这些现有抽象扩展，而不是另起一套完全独立的数据结构。

---

## 七、验收口径

本次优化建议至少满足以下体感目标：

1. 切换到包含超长 assistant 消息的会话时，首屏不再出现数秒级冻结。
2. 大消息首次进入可视区时，卡顿明显下降，不再出现“窗口像假死”的感觉。
3. 在最近 50 条消息中包含多条长 Markdown / 代码块时，滚动保持连续，不出现大面积掉帧。
4. 流式回复中的尾部滚动、active turn 对齐、jump-to-bottom 仍然可用。

---

## 八、最终建议

这次问题的主解排序建议如下：

1. **先做单条重消息降载**
2. **再做聊天区动态高度虚拟滚动**
3. **倒置滚动暂不作为第一优先级**

一句话总结就是：

- **你提的两个方向里，虚拟滚动是对症的，倒置滚动不是主药；但仅做虚拟滚动还不够，还需要顺手解决“单条超大消息一次性完整渲染”的问题。**
