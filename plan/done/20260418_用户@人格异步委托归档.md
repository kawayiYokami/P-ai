# 用户 @ 人格异步委托归档（2026-04-18）

## 1. 背景

本轮目标是把聊天输入区的 `@` 从“临时前端状态”升级成真正可落消息、可回放、可委托、可被模型理解的用户级异步委托机制。

用户在这轮里反复强调了几个关键边界：

- `@` 不是正文文本解析，而是用户显式选中的结构化 mention
- `@` 的候选展示人格，不展示部门；但只有存在部门归属的人格才允许被 `@`
- `@` 不是切换当前主回答者，而是“用户发起的异步委托”
- 用户委托和 LLM 内部 delegate 不是一回事
- 有 `@` 时，当前会话负责部门不应继续走主回答链
- 委托结果不该再依赖 `history_flushed`，而应该像异步追加消息一样直接写回前端

用户已在实际界面上连续验证多轮，本归档按最终落地结果收口。

## 2. 实际落地

### 2.1 输入区 mention 交互

聊天输入区已支持结构化 mention：

- 输入裸 `@` 时弹出人格候选浮层
- 候选按人格去重，只保留存在部门归属的人格
- 候选浮层与工具栏头像都支持 toggle：
  - 未选时添加
  - 已选时取消
- mention 不再解析正文文本，也不会因复制/粘贴正文触发

同时，已选 mention 与候选列表的视觉规则已统一：

- 候选列表使用浮层弹出，不再顶开输入面板
- 已选 mention 在输入框上方以胶囊形式显示
- 候选项已选状态通过头像角标表达，而不再依赖整行背景色
- 工具栏头像也同步显示 `@` 角标；当前会话负责人格在存在 mention 时显示“禁”角标，并改为灰度头像

### 2.2 发送语义分流

发送链路已正式拆成两条：

1. 无 mention
- 继续走原有 `send_chat_message`
- 当前会话负责部门正常回答

2. 有 mention
- 前端直接调用独立命令 `send_user_mention_message`
- 当前会话只写入用户消息
- 不再激活主回答链
- 只为被 mention 的目标触发用户级异步委托

这一点是本轮最关键的语义修正之一。最终行为已经从“主路径 + 关激活”改为“直接走单独用户委托路径”。

### 2.3 人格到部门的映射

`@` 的候选展示人格，但运行时实际执行部门按以下规则解析：

- 每个人格映射到其绑定部门中的“配置顺序第一项”
- 该映射在发送时固化为快照
- 发送后不再因为部门顺序变化或绑定关系变化而漂移

因此，运行时真正参与执行的是“发送时解析出的首部门”，而不是动态重算的当前部门。

### 2.4 用户消息元数据持久化

mention 已经正式进入消息记录，而不是只存在发送时参数里。

当前用户消息会把解析后的 mention 快照写进消息元数据，用于：

- 消息气泡渲染
- 撤回恢复
- 重新生成恢复
- 后续 prompt 构建

也因此，撤回这条用户消息时，输入框里的 mention 挂件会一起恢复，不再只恢复正文。

### 2.5 用户气泡与草稿显示

用户消息的 mention 显示方式已收口为：

- 不再单独挂在正文上方
- 直接作为正文前缀显示

例如：

- `@fairy 你来，他们好笨……`

这条规则同样适用于：

- 本地发送草稿
- 历史消息渲染
- 撤回恢复后的再次发送

从而保证发送前后视觉一致，不再出现“草稿和正式消息显示不一致”的情况。

### 2.6 发给模型的内容

模型可见的 mention 语义已按最简方案接入：

- 发送给 LLM 的用户消息正文前缀直接拼接 `@A,@B`
- 不增加额外复杂 JSON 块
- 不再把内部委托字段裸暴露给模型

同时，本轮也明确清理了错误实现边界：

- 通用消息渲染函数只负责普通消息内容渲染
- mention 前缀只允许在“请求体构建层”处理
- 内部委托元数据（如 `targetAgentId` / `targetDepartmentId`）已经从模型可见文本中移除

### 2.7 委托结果写回链路

本轮把委托结果回写机制也一起收口了。

之前委托结果依赖：

- 写回会话
- 再走 `scheduler -> history_flushed`

这会导致一个老问题在 mention 路径下被明显放大：

- 后端 delegate 已完成
- 但前端没有及时看到结果

现在所有委托结果都改为：

1. 直接写回原会话
2. 成功后发出 `conversation-message-appended` 事件
3. 前端直接 append 这一条消息

这条规则对两类委托统一生效：

- 用户 `@` 异步委托
- LLM delegate 工具委托

也就是说，委托结果现在不再依赖整段历史刷新，而是统一走“异步追加消息”语义。

### 2.8 委托失败与边界行为

用户明确要求：

- 失败异常应该直接作为委托结果返回
- 不要静默吞掉
- 不要把失败包装成别的提示流

最终实现已按此收口：

- 目标部门不存在 / 人格解绑 / 模型失败 / 超时 / 运行失败
- 都尽量作为该条委托结果返回
- 会话删除或归档时，不再补写回已失效会话

同时，原先“委托写回失败静默吞掉”的问题也已补日志，不再黑盒消失。

## 3. 与原始目标相比的关键修正

本轮中途有几次实现方向被用户明确纠正，最终实际落地和最初草案相比，关键修正如下：

1. `@` 不是切换当前回答者，而是用户级异步委托。
2. 有 mention 时，不再让当前会话负责部门参与主回答。
3. 委托结果不再走 `history_flushed`，而统一走“追加一条消息”。
4. mention 必须进消息元数据，并随撤回/重放一起恢复。
5. 发给模型的 mention 信息只保留人类可读前缀，不暴露内部委托字段。

## 4. 当前结果

本轮已完成：

- 输入区结构化 mention 挂件与候选浮层
- 工具栏头像 toggle mention
- mention 候选按人格去重、按首部门解析
- 有 mention 时独立走用户委托发送命令
- 用户消息元数据持久化 mentions
- 用户草稿 / 历史消息统一前缀显示 mention
- 撤回 / 重新生成时恢复 mention
- 发给模型的用户消息前缀拼接 `@A,@B`
- 清理内部 `targetAgentId` / `targetDepartmentId` 文本泄漏
- 所有委托结果统一改为 append-message 事件直推前端
