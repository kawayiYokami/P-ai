# 记忆页记忆备份导入导出重构计划

## 1. 背景

当前 `easy_call_ai` 记忆页已经支持导入本项目导出的扁平 `memories` JSON，但现在目标已经改变：要统一记忆迁移协议，直接采用 Angel Memory 自动备份使用的中央记忆 JSON 结构，例如：

- `memory_center/backups/memory_backup_*.json`

该备份本质上与当前项目记忆架构一致，只是导出时按表展开为：

- `records`
- `global_tags`
- `memory_tag_rel`

因此本次不做底层数据库直连，也不做通用迁移页，而是同时完成两件事：

1. 本项目记忆导出改为与 Angel 相同的三表展开 JSON
2. 在“记忆页”新增两个独立入口：**导入记忆备份** 与 **导出记忆备份**

## 2. 目标

实现两类结果：

1. 本项目记忆导入导出统一为 Angel 同款 JSON 结构
2. 提供一条面向用户的显式导入导出链路：

1. 用户在记忆页点击“导入记忆备份”
2. 先打开导入弹窗，再选择备份 JSON 文件
3. 预览框展示：
   - 读取到多少条记忆
   - 识别到哪些 `memory_scope`
   - 每个 `memory_scope` 对应的人格选择
   - 随机抽查 10 条记忆
4. 用户确认作用域映射后再正式导入
5. 用户在记忆页点击“导出记忆备份”
6. 先打开导出弹窗，读取当前内部记忆域并勾选后再导出

## 3. 非目标

- 不直接读取 Angel Memory 的 SQLite 数据库
- 不迁移 `tantivy_indexes/`、`maintenance_state.json`、其他索引层文件
- 不改动当前项目底层记忆存储架构
- 不做自动 watch 或自动同步
- 不兼容旧扁平 `memories` JSON 导入格式

## 4. 用户体验设计

### 4.1 入口位置

放在记忆页配置区下方，作为两个独立卡片。

按钮文案建议：

- `导入记忆备份`
- `导出记忆备份`

### 4.2 交互流程

#### 第一步：打开导入弹窗

用户先打开导入弹窗，再主动选择 `memory_backup_*.json` 文件。

#### 第二步：后端预解析

后端读取并校验 Angel 同款备份 JSON，恢复出标准化记忆数组，但此时**不落库**。

#### 第三步：弹窗预览

弹窗展示：

- 总记忆条数
- 作用域列表
- 每个作用域对应的人格下拉框
- 抽查样本 10 条

每条样本建议展示：

- `memory_scope`
- 记忆类型
- judgment
- tags
- created_at

#### 第四步：确认导入

用户点击确认后，才将预解析结果按 `memory_scope -> 人格` 映射导入到本地记忆库。

## 5. 数据适配方案

### 5.1 统一迁移格式

统一后的导入导出 JSON 结构：

- `schema_version`
- `exported_at`
- `records`
- `global_tags`
- `memory_tag_rel`

本项目导出后也使用这套结构，不再继续导出扁平 `memories`。

### 5.2 还原逻辑

后端导入预解析时按以下步骤还原：

1. 读取 `global_tags`，构建 `tag_id -> tag_name`
2. 读取 `memory_tag_rel`，构建 `memory_id -> tags[]`
3. 遍历 `records`
4. 为每条记录补上 `tags[]`
5. 转成当前项目可导入的 `MemoryEntry`

后端导出时做反向展开：

1. 遍历当前记忆
2. 收集去重后的 `global_tags`
3. 生成 `memory_tag_rel`
4. 输出统一三表结构 JSON

### 5.3 作用域与人格映射

本次导入不是“全部导入到一个人格”，而是：

- 先按源文件中的 `memory_scope` 分组
- 再由用户为每个 `memory_scope` 选择目标人格

规则：

- 若任一 `memory_scope` 未选择人格，不允许确认导入
- 导入时将该作用域下的记录写为对应人格私有记忆
- 导入落库时：
  - `owner_agent_id = 目标人格 id`
  - `memory_scope = 目标人格作用域`

### 5.4 导出作用域规则

本项目导出时，`memory_scope` 不再简单沿用内部默认值，而是按人格作用域输出：

- 若记忆属于某个人格私有记忆，则 `memory_scope` 导出为该人格作用域
- 当前阶段“人格作用域”直接使用人格名
- 若记忆不属于任何人格，则保留为 `public`

### 5.5 当前阶段的字段策略

本轮优先保证导入导出协议统一与字段语义对齐：

- 必须保留：
  - `id`
  - `memory_type`
  - `judgment`
  - `reasoning`
  - `tags`
  - `created_at`
  - `updated_at`

对于 Angel 备份中存在、且统一导出协议也应稳定保留的字段：

- `strength`
- `is_active`
- `memory_scope`
- `useful_count`
- `useful_score`
- `last_recalled_at`
- `last_decay_at`

本轮要求：

1. 预解析时完整保留这些字段
2. 本项目导出时也完整写出这些字段
3. 不再以旧扁平结构作为兼容目标

## 7. 抽查样本策略

为了避免抽样结果太偏，第一版建议：

- 若总条数 <= 10，全部展示
- 若总条数 > 10：
  - 先固定取前 3 条
  - 再从剩余中随机取 7 条

这样既能看到文件头部是否正常，也能覆盖部分随机样本。

## 8. 风险与注意点

### 8.1 类型映射风险

若 Angel 备份中的 `memory_type` 与本项目当前允许值不完全一致，需要在预解析阶段做显式映射与报错提示。

### 8.2 时间格式风险

若 Angel 使用 epoch/float 时间，而当前项目使用 ISO 字符串，需要统一转换。

### 8.3 私有记忆导入边界

作用域映射导入必须走现有“按人格导入”语义，避免把不同作用域的记忆混写到错误人格或误写为全局记忆。

### 8.4 大文件体验

预解析与样本抽查应只做一次，不应在前端重复全量解析大文件。
